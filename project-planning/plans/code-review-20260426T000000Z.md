## Critical Findings

### C-1. `useCacheEntry` does not subscribe to cache mutations — cards permanently stuck on "Loading…"

**File:** `site/next-app/lib/cache/useCacheEntry.ts:34-43` (pre-fix)
**Category:** correctness
**Severity:** critical

**What's wrong**
The hook `useCacheEntry` reads from a singleton `Map<CacheKey, PageDerivedState>` via `getEntry(key)` inside `useMemo([id, version])`. When `prefetchPageUrls` resolves milliseconds after the first render and writes the slot via `setEntry(...)`, no React re-render fires because the Map mutation is invisible to React. The cards render on first `pages.context.onSuccess`, see `state: undefined`, render the "Loading…" tooltip + `aria-disabled="true"` — and stay there forever. The header comment in the same file documents the missing subscription pattern but the implementation never landed it.

In the existing test suite this defect was masked because:
- the a11y test (`__tests__/a11y.test.tsx`) pre-seeds the cache via `setEntry` BEFORE rendering, so the cards see resolved state on first paint;
- the prefetch wiring tests (`marketplace.prefetch.test.tsx`) assert that `prefetchPageUrls` is called but never render a card to observe the post-resolve state;
- the per-card unit tests stub `useCacheEntry` directly and never exercise the real cache singleton.

In the actual Sitecore Pages editor — first install, real SDK — every action card would be permanently disabled with the "Loading…" tooltip until the user toggles the theme or navigates pages. v0.1 would ship broken.

**Suggested fix (applied)**
Add `subscribe(listener)` + `getVersion()` to `lib/cache/store.ts`. Every mutation (`setEntry` / `patchEntry` / `clearAll`) bumps a monotonic counter and notifies listeners. Convert `useCacheEntry` to drive re-renders via `useSyncExternalStore(subscribe, getVersion, () => 0)`. The server snapshot is `0` so SSR/jsdom paths stay correct. New regression test `lib/cache/useCacheEntry.test.tsx` exercises the post-paint mutation flow.

## Major Findings

### M-1. `ShareLinkSplit` primary `aria-label` flips during morph — `Alt+S` silently breaks for 1500ms

**File:** `site/next-app/components/quickcopy/ShareLinkSplit.tsx:209-212` (pre-fix)
**Category:** correctness / accessibility
**Severity:** major

**What's wrong**
The panel-level `Alt+S` shortcut dispatches a synthetic click via accessibility-name substring match in `app/panel/page.tsx:37-45`:
```ts
root.querySelector(`button[aria-label*="${suffix}"]:not([aria-disabled="true"])`)
```
The Share Link primary button's `aria-label` was state-conditional:
```ts
const primaryAriaLabel = mode === "copied" ? "Copied" : "Copy Share Link as Markdown — shortcut Alt+S";
```
During the 1500ms post-success morph, the `aria-label` flipped to `"Copied"` — losing the `"shortcut Alt+S"` suffix the panel's selector relies on. A user pressing `Alt+S` twice in quick succession would copy once and have the second press silently swallowed because the selector found no match. ActionCard cards (which keep `aria-label` stable as a prop) do NOT have this defect.

**Suggested fix (applied)**
Pin the primary's `aria-label` to `"Copy Share Link as Markdown — shortcut Alt+S"` regardless of `mode`. Visible label still flips via `primaryLabel`. Mirrors the ActionCard contract. New test (`ShareLinkSplit.test.tsx`, "primary aria-label is stable across the 'Copied' morph window") asserts the aria-label is identical before and after the morph.

### m-1. `MarketplaceProvider` couples `client.destroy()` to the `[client]` effect cleanup
**File:** `site/next-app/components/providers/marketplace.tsx:169-175`
**Category:** architecture (latent)
The pages.context subscription effect's cleanup also calls `client.destroy()`. The dependency array is `[client]`, and since `client` only goes `null → instance` once in current code paths the cleanup only runs on unmount — so the existing test (`marketplace.test.tsx:150-167`) passes. But destroying the SDK from the same effect that subscribes to a single key is a fragile coupling: any future change that re-creates `client` (e.g. retry-on-init-error, environment swap) would tear down the SDK between subscribe and re-subscribe. Cleaner pattern: keep `destroy()` in a top-level `useEffect(() => () => client?.destroy(), [client])` separate from the subscription effect, OR move it to a separate effect explicitly keyed to mount/unmount. Defer until there's a reason to touch this code.

### m-2. `ItemIdCard` and `PageTitleCard` use the copied value as their cacheKey for sticky-error scoping
**Files:** `site/next-app/components/quickcopy/ItemIdCard.tsx:44`, `PageTitleCard.tsx:38`
**Category:** code quality / spec interpretation
ADR-0009 says sticky errors clear "when cache key changes". The four URL cards use `${pageId}:${version}` (the canonical cache key). ItemIdCard and PageTitleCard use the trimmed value being copied (`id || null`, `title || null`). Functionally close because pageId determines the displayed id/title — but if a user encounters a clipboard rejection on Page A, navigates to Page B with the same Page Title, the error remains stuck on the Page Title card alone. Probably acceptable per the broader spec ("error sticks until cache key changes"), and the alternative would require threading the canonical pageInfo cacheKey into both cards. Worth a tiny refactor when next touching either card.

### m-3. Regression audit treats `as never` casts in `prefetch.ts` as exempt; documented but the rule is loose
**File:** `site/next-app/lib/regression-audit.test.ts:190-214`
**Category:** test quality
The audit forbids `as string` and `as any` outright but explicitly does not flag `as never`. The justification (header comment, point 4) is that the 3 `as never` casts in `prefetch.ts:131,141,147` are on the SDK-call **request payload** (a Phase-1 workaround for the narrow generic on `client.query`), not on return values. That's a sound exception but it's a single-line allowance for a single file. If `as never` ever leaks into a return-value-cast, the audit won't catch it. Optional tightening: switch the audit to forbid `as never` everywhere and explicitly whitelist the three call sites in `prefetch.ts` by file+regex. Low priority; the casts were green-light by QA.

### m-4. `ShortcutLegend` declares both `<ul>` and `role="list"`
**File:** `site/next-app/components/quickcopy/ShortcutLegend.tsx:26-28`
**Category:** code quality
`<ul>` has implicit `role="list"`. The explicit role does no harm (and could survive a `list-style: none` Safari accessibility-tree quirk) but it's redundant. Drop or keep — taste call.

### m-5. `useCopyAction.onActivate` rapid-clicks during the morph window re-fire the clipboard write
**File:** `site/next-app/components/quickcopy/useCopyAction.ts:73-93`
**Category:** code quality / spec interpretation
Per `state="copied"` being treated as `isInteractive=true` in `ActionCard.tsx:79`, a click during the 1500ms morph fires `onActivate` again, which awaits `copyTextToClipboard` and resets the timer. B-062 mandates "Subsequent clicks within the morph window reset the timer" — currently they reset the timer AND re-issue the clipboard write. That's likely fine (the clipboard ends up with the same value) and matches the test, but if some browser surface shows a flash from a rapid double-write that's where to look.
