### Major (should fix before release)

#### M-1. `useShortcuts` does not suppress `Shift+Alt+L`/`Shift+Alt+S` etc. — contradicts ADR-0008 § 4 and the test prompt
**File:** `hooks/useShortcuts.ts:87-100`
**Category:** correctness / spec deviation / cross-platform UX

**What's wrong**
ADR-0008 § "Listener implementation contract" item 4 specifies: *"Match on `event.altKey`, `event.ctrlKey`, **`event.shiftKey`**, `event.metaKey`, and `event.key.toLowerCase()`."* The current implementation (`useShortcuts.ts` lines 87-100) only requires `altKey === true && ctrlKey === false && metaKey === false` — `event.shiftKey` is never read. As a result, `Shift+Alt+L` (and the four sibling Shift-augmented variants) FIRE the live/preview/item/title/share handlers, which is exactly the behavior the ADR forbids. The test suite never exercises a `shiftKey: true` keyboard event — `useShortcuts.test.ts` only covers `Ctrl+Alt+L`, `Meta+Alt+L`, and `Alt+X`. The hook's own JSDoc at lines 22-23 even claims "`Ctrl+Alt+L`, `Meta+Alt+L`, and Shift-anything are treated as not-our-combo" — which is a documentation/implementation drift.

**Why this matters in practice:**
- On macOS, `Option+L` (`Alt+L`) inserts `¬` while `Shift+Option+L` (`Shift+Alt+L`) inserts `Ò`. A Pages-editor user on macOS who fat-fingers `Shift+Alt+L` while trying to type `Ò` into a content field that happens to be inside the QuickCopy iframe — for example a future hypothetical inline preview — would have the live URL silently copied to clipboard instead of `Ò` being inserted. The current INPUT/TEXTAREA guard catches typing INTO an editable element, but this contract is brittle.
- The PRD's keyboard-shortcut conflict surface (T038) only tests `Alt+<letter>`, never the Shift variants. A real Sitecore Pages editor binding on `Shift+Alt+L` would conflict invisibly.
- The ADR says it. The hook's own comment says it. The implementation doesn't. Pick one source of truth and align the other two.

**Suggested fix**
Add `if (event.shiftKey) return;` after the existing `metaKey` check on line 90. Add `'Shift+Alt+L does NOT fire'` and one for `Shift+Alt+S` to `useShortcuts.test.ts` (these will RED until the line is added). Update ADR-0008 status note only if the team decides to relax the contract instead — but the ADR is currently authoritative.

#### M-2. `<MarketplaceProvider>`'s loading state uses `aria-live="polite"` outside `StatusLiveRegion.tsx` — regression audit doesn't catch it because the audit only scans `components/quickcopy/`
**File:** `components/providers/marketplace.tsx:228-229`
**Category:** spec deviation (potential) / regression-audit gap

**What's wrong**
ADR-0009 § "What this ADR explicitly excludes from v0.1": *"No `aria-live` announcement of errors (would be intrusive — five buttons each potentially erroring would create announcement spam)."* The Provider's loading state at lines 228-229 renders `<div role="status" aria-live="polite">Connecting to Sitecore Marketplace…</div>`. This is for a SUCCESS-style loading announcement (not an error), and ADR-0009 only forbids aria-live for errors — so the **runtime behavior is arguably acceptable**. However, the regression audit at `lib/regression-audit.test.ts:175-187` (rule 3) scans `components/quickcopy/` only and would not flag a future regression that adds `aria-live` to the per-button error path inside `components/providers/`. If a future change adds an error-state aria-live outside `components/quickcopy/`, the audit will not catch it.

**Why this matters in practice:**
- The audit's stated goal (header comment, item 3) is "errors are visual-only — there must be no error-state aria-live region." If that protection is meant to be repo-wide, the audit's scan path is too narrow.
- The current Provider use is a *loading* aria-live, not an *error* aria-live, so the run-time risk today is zero. This is a future-proofing finding, not a current breakage.

**Suggested fix**
Either (a) widen the audit's `roots` array in test 3 to include `components/providers/` (and assert `StatusLiveRegion.tsx` is the only allowed file), OR (b) add a new audit rule that explicitly permits the Provider's loading-state line by file+regex, OR (c) accept the current scope as intentional and add a comment to that effect in `regression-audit.test.ts` so future readers don't widen unilaterally. Lowest-cost option: widen the scan path and explicitly allow `components/providers/marketplace.tsx`.

### Minor (acceptable risk)

#### m-1. Item ID and Page Title cards use the trimmed copy value as their sticky-error cacheKey, not the canonical page cache key
**Files:** `components/quickcopy/ItemIdCard.tsx:44`, `components/quickcopy/PageTitleCard.tsx:38`
**Category:** code quality / spec interpretation
This is the same finding the code-review documented as `m-2`. ADR-0009 says sticky errors clear "when cache key changes". Live and Preview cards key the sticky error on `${pageId}:${version}` (canonical). Item ID and Page Title cards key on `id || null` and `title || null` respectively. Functional behavior is close (page id determines the displayed id/title), but if a user encounters a clipboard rejection on Page A, navigates to Page B with the same Page Title text, the error would remain stuck on the Page Title card even though the cache key changed. The probability of this collision is low; the fix is small but not free. Defer until next touch unless field reports surface it.

#### m-2. `prefetch.ts` uses `as never` casts on SDK request payloads; documented but the audit's exception is broad
**File:** `lib/url-resolver/prefetch.ts` (per code-review m-3)
**Category:** test quality
Same as code-review m-3. The audit forbids `as string` and `as any` outright but does not flag `as never`. The justification (regression-audit.test.ts header item 4) is that the three `as never` casts are on the SDK-call **request payload** (Phase-1 workaround for the narrow generic on `client.query`), not on return values. Sound exception, but the rule is loose — if `as never` ever leaks into a return-value cast, the audit won't catch it.

#### m-3. `<ShortcutLegend>` declares both `<ul>` and `role="list"`
**File:** `components/quickcopy/ShortcutLegend.tsx:26-28` (per code-review m-4)
**Category:** code quality
`<ul>` has implicit `role="list"`. Explicit role does no harm (and could survive a `list-style: none` Safari accessibility-tree quirk). Drop or keep — taste call.

#### m-4. `useCopyAction.onActivate` re-issues the clipboard write on rapid re-click during the 1500ms morph window
**File:** `components/quickcopy/useCopyAction.ts:73-93` (per code-review m-5)
**Category:** code quality / spec interpretation
Same as code-review m-5. B-062 mandates "Subsequent clicks within the morph window reset the timer." The implementation resets the timer AND re-issues the clipboard write because `state="copied"` is treated as `isInteractive=true` in `ActionCard.tsx:79`. Acceptable for v0.1 — the clipboard ends up with the same value — but worth noting if a browser surface flashes on rapid double-write.

#### m-5. No `<QuickCopyPanel>` integration test file actually exists despite § 10 T036a defining one
**File:** missing — task-breakdown § 10 T036a (`components/quickcopy/QuickCopyPanel.test.tsx`)
**Category:** coverage gap (intentional — T036b was simplified to compose directly in `app/panel/page.tsx`)
The task breakdown's § 10 T036a defines an integration suite at `components/quickcopy/QuickCopyPanel.test.tsx`. The implementation's panel composition lives in `app/panel/page.tsx` rather than a separate `<QuickCopyPanel />` component, so there is no `QuickCopyPanel.test.tsx`. The functional behaviors T036a was designed to cover are partially distributed across the per-card test files + `useShortcuts.test.ts` + `__tests__/a11y.test.tsx`, but the eleven canonical T036a integration scenarios (B-135, B-136, B-147 + tab-walk, version-bump-clears-error, single-card-error-isolated) are NOT all directly asserted. The accessibility smoke at `__tests__/a11y.test.tsx` does pre-seed the cache and run jest-axe in both themes, which covers B-148/B-149. **Recommended action:** either add a small `app/panel/__tests__/page.test.tsx` integration suite that mounts the route component with stubbed Provider for the missing scenarios, OR explicitly mark T036a as "covered piecewise by per-component tests" in the test contract. The risk of NOT having it is the integration scenarios from § 10 T036a (cache-key bump clears error across cards, single-fetch-rejection isolates one card, tab order theme-toggle → Live → Preview → Item → Title → Share-primary → Share-caret) are not regression-pinned. T038 (real-portal smoke) is the only remaining safety net — which is explicitly manual.

#### m-6. Bundle-size NFR-008 verification is a static comment in `next.config.mjs`, not a programmatic gate
**File:** `next.config.mjs:6-24`
**Category:** test quality / regression risk
NFR-008 sets a ≤150 KB gzipped client target. The current first-load gz size for `/panel` is ~148 KB per the comment in `next.config.mjs` (verified manually during this Phase 1: root-main 4 chunks gz = ~120 KB + polyfill ~39 KB + panel-specific entry ~28 KB → effective ~148 KB rounded). The runbook's verification checklist mentions "Bundle size within NFR-008 target" as a manual sweep step. There is no automated guard — adding a single icon library or two heavy Blok primitives in v0.2 would silently push the bundle over the target. Optional addition: a tiny `bundle-size.test.ts` that reads `.next/static/chunks/*.js`, gzips them, sums root-main + panel, and asserts `< 150 * 1024`. Defer until after v0.1 ships.
