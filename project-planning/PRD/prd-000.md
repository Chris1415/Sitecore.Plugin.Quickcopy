## 1. Executive Summary

**QuickCopy** is a Sitecore Marketplace Page Context Panel app that gives marketers and content authors one-click copy buttons for the five pieces of page metadata they paste into Teams, Slack, email, and Jira most often: **Live URL, Preview URL, Item ID, Page Title, and a Share Link** (Page Title + URL composite, available in Markdown and plain-text formats).

The app lives in the `xmc:pages:contextpanel` extension point, reads live page context from the Marketplace SDK `pages.context` query, and makes authoritative Agent-API calls for URL freshness and publication status. The UI is built with **Blok**, defaults to dark mode with a light-mode toggle, and is fully keyboard-operable with a visible shortcut legend.

QuickCopy is the first-party, purpose-built Marketplace app for the "quickly share a page" use case. It does not attempt to be a dashboard, a publishing tool, or a content-management surface — it is five buttons and a theme toggle, built with honesty (disabled state for unpublished Live URL, persistent error state for failed API calls) and polish (dark-mode default, Blok design system).

**Effort band:** XS–S (hours to a day of focused work, subject to Live URL composition complexity).

## 2. Problem Statement

Marketers and content authors working in the Sitecore Pages editor share page references dozens of times per day — review links in Slack, bug tickets in Jira, release-note entries, handoff messages to colleagues. Today this requires:

- Hunting for the page URL in the browser address bar (often grabs the Pages editor chrome URL instead of the actual content page).
- Opening a separate tab to find the live URL if the page is already published.
- Digging into page-detail views to find the Item ID when filing a developer ticket.
- Manually typing "Page Name — https://…" combinations for rich-context pasting in Slack, Teams, Jira, or Notion.

Each of these actions is a small context switch, but aggregated across a team they represent meaningful friction and a real source of copy-paste errors (wrong URL, wrong environment, stale link). The information needed is already present in the `pages.context` SDK query or a single Agent-API round trip — it just needs a one-click surface beside the page the user is already editing.

## 3. Goals and Success Metrics

### Product goals

1. **Eliminate the "hunt for the URL" friction** — the user never leaves the Pages editor to copy a page reference.
2. **Serve both solo-paste and cross-functional-paste needs** — five button shapes cover marketer-to-marketer (URLs, title) and marketer-to-engineer (Item ID) handoffs in one panel.
3. **Be honest by default** — the app never hands the user a broken link. Unpublished Live URLs are disabled with an explanatory tooltip; API failures show a persistent error state, not a silent success.
4. **Dogfood the agentic pipeline at XS scope** — prove the `/rubber-ducky → /create-prd → /architect → /task-breakdown → /implement → /ship` flow handles small real apps without ceremony overhead.

### Success metrics (post-ship)

- **Usage:** at least one copy action per session in ≥50% of QuickCopy-panel-opened Pages editor sessions, within 2 weeks of ship. *(Measurement deferred to v0.2 — v0.1 ships with zero telemetry, see §14.)*
- **Correctness:** zero reports of "copied the wrong URL" or "pasted a broken link" for the first 30 days post-install, per stakeholder feedback channels.
- **Install simplicity:** install-to-first-successful-copy in ≤3 minutes for a marketer with Marketplace admin access, measured by stakeholder dogfooding session.
- **Pipeline DX (internal):** full `/rubber-ducky → /ship` cycle completes in ≤1 working day with no stage blocked by pipeline ambiguity.

### Explicit non-goals

- Building a dashboard, widget, or full-screen surface (panel only).
- Surfacing developer metadata (Item Path, Template ID/Name, Version) — see §5 Out of Scope.
- Editing page data — QuickCopy is a read-only tool.
- Publishing or unpublishing actions — those are adjacent apps (Publish Impact Radar, backlog #2).

## 4. Target Users

### Primary persona — **Marketer in the Pages editor**

- **Role:** Content author, campaign lead, or marketer working directly in Sitecore Pages.
- **Daily context:** Has the page open in the editor, knows what page they're referencing, needs to share it right now with a teammate via Slack / Teams / email / Jira.
- **Technical depth:** Low–medium. Comfortable in the Pages UI; not comfortable with GraphQL, tree paths, or template GUIDs.
- **Keyboard comfort:** Mixed. Some power users want shortcuts; casual users use the mouse. Both must work.
- **Theme preference:** Varied; the default must not hurt any of them. Dark is the first-class default, light is a full-quality toggle.

### Secondary persona — **Marketing ops / PM / QA reviewer**

- **Role:** People who coordinate across marketers and engineering. They write tickets, release notes, approval requests.
- **Daily context:** Intermittent Pages editor users. When they're in Pages, they usually need to pass a page reference *outside* — to Jira, to GitHub, to Confluence.
- **Key need:** The **Share Link** button, which produces a title-plus-URL composite ready for rich paste into markdown-aware tools or plain-text channels.

### Non-users (explicit)

- **Developers and Sitecore implementers:** They have Content Editor, XM Cloud deploy pipelines, and Experience Edge GraphQL. QuickCopy deliberately excludes the fields they need (Item Path, Template ID, Version) to stay marketer-first.

## 5. Product Scope

### In Scope

**Panel and surface**
- Extension point: `xmc:pages:contextpanel` (Pages editor right-hand sidebar).
- Panel renders inside the standard Marketplace iframe; no custom hosting.
- Reacts live to `pages.context` updates (page navigation inside Pages re-renders buttons immediately).

**Five copy actions**
1. **Copy Live URL** — full absolute URL of the published page.
2. **Copy Preview URL** — full absolute preview URL.
3. **Copy Item ID** — GUID of the current page item.
4. **Copy Page Title** — the page's display name.
5. **Copy Share Link** — composite of Page Title + URL, split-button dropdown offering **Markdown** and **Plain Text** formats (see §7 FR-005).

**UI and interaction**
- Blok UI framework throughout (Button, IconButton, Tooltip, Toggle, Kbd-or-equivalent).
- Dark mode is the default theme; a theme switcher toggles light mode.
- Theme preference persists locally across panel reloads.
- All five actions have keyboard shortcuts with a visible legend in the panel.
- Success feedback is an in-place button morph ("Copied!") for 1500ms, then returns to idle. **No toast notifications.**
- Failure feedback is a **persistent** error state on the affected button until the page context changes (see §7 FR-007 for full behavior).

**Data sources**
- `pages.context` SDK subscription for page ID, display name, language, site context.
- Agent API `GET /api/v1/pages/{id}/preview-url` for the authoritative Preview URL.
- Agent API `GET /api/v1/pages/{id}/live` for publication status gating Live URL.
- Sites API `GET /sites/{siteId}/hosts` (or SDK `/xmc` equivalent) for Live URL host resolution — only called when live gate passes.

**Reused assets**
- Clipboard-write pattern from `products/pageshot/` (proven iframe-safe `navigator.clipboard.writeText` handling).

### Out of Scope

- Developer/ops fields: Item Path, Template ID, Template Name, Version, Site ID, Site Name, Language code.
- Telemetry, analytics, or usage counters in v0.1.
- Multi-site picker UI — the current `pages.context` site is used silently.
- Multi-language picker UI — the current `pages.context` language is used silently.
- Batch "Copy all" actions.
- Customization of Share Link templates (fixed formats only).
- Dashboard widget, standalone, or full-screen variants.
- Publishing, unpublishing, or edit actions on the page.
- Onboarding wizard or feature tour.
- Localization of the app's own UI chrome — English only in v0.1.

## 6. User Stories

### US-001 — Marketer copies a Live URL for sharing a published page

**As a** marketer viewing a published page in the Pages editor,
**I want to** click one button to copy the live public URL to my clipboard,
**So that** I can paste it into Slack/Teams/email without leaving Pages.

**Acceptance criteria:**
- Given the page is published to Edge, when I click **Copy Live URL**, the button morphs to "Copied!" for 1500ms and my clipboard contains the full absolute URL including scheme and host (e.g. `https://www.example.com/products/spring-campaign`).
- Given the page is **not** published, the **Copy Live URL** button is disabled; hovering shows the tooltip **"Not published to Edge yet — publish the page first"**.
- Given the Live URL API call fails (network, 5xx, auth), the button shows a persistent ❌ error state with the tooltip **"Couldn't fetch Live URL — try switching pages or reloading"**, and remains in error until the `pages.context` page ID changes.

### US-002 — Marketer copies a Preview URL for draft review

**As a** marketer who has just finished editing a page,
**I want to** click one button to copy the authoritative preview URL,
**So that** I can send it to a reviewer to inspect the draft before publication.

**Acceptance criteria:**
- When I click **Copy Preview URL**, the app fetches the URL from `GET /api/v1/pages/{id}/preview-url` (not from the possibly-stale `pageInfo.url`), the button morphs to "Copied!" for 1500ms, and my clipboard contains the full absolute preview URL.
- The fetched URL is cached per page-item for the duration of that `pages.context` selection (a second click does not re-fetch).
- Given the Preview URL API call fails, the button shows a persistent ❌ error with tooltip **"Couldn't fetch Preview URL — try switching pages or reloading"** until the `pages.context` page ID changes.

### US-003 — Marketer copies Item ID for a developer ticket

**As a** marketer filing a Jira ticket about a page issue,
**I want to** click one button to copy the page's Item ID (GUID),
**So that** I can paste it into the ticket for the developer without switching views.

**Acceptance criteria:**
- When I click **Copy Item ID**, the button morphs to "Copied!" for 1500ms and my clipboard contains the GUID from `pageInfo.id` with no wrapping braces and no whitespace.
- The action completes with no HTTP round trip.
- Given `pageInfo.id` is unexpectedly missing, the button is disabled with tooltip **"Page context not ready — wait or reload"**.

### US-004 — Marketer copies the Page Title

**As a** marketer referring to a page in a longer message,
**I want to** click one button to copy the page's display name as plain text,
**So that** I can type "Fix bug on **Spring Campaign Landing Page**" without retyping the name.

**Acceptance criteria:**
- When I click **Copy Page Title**, the button morphs to "Copied!" for 1500ms and my clipboard contains the string from `pageInfo.displayName` (not `pageInfo.name`, which is the machine name).
- No HTTP round trip.
- Given `pageInfo.displayName` is missing or empty, fall back to `pageInfo.name`; if both are missing, disable the button with tooltip **"Page context not ready — wait or reload"**.

### US-005 — PM/ops reviewer copies a Share Link for a handoff

**As a** marketing ops coordinator preparing a release-note entry,
**I want to** click one button to get a page-title-plus-URL composite in my preferred format,
**So that** I can paste a rich, scannable reference into Notion or Slack in one move.

**Acceptance criteria:**
- **Copy Share Link** is a split-button with a caret. Clicking the primary button copies the **default format (Markdown)**. Clicking the caret opens a dropdown with two options: **"Copy as Markdown"** and **"Copy as Plain text"**.
- **Markdown format:** `[Display Name](Live URL)` — e.g. `[Spring Campaign Landing Page](https://www.example.com/products/spring)`.
- **Plain-text format:** `Display Name — Live URL` — e.g. `Spring Campaign Landing Page — https://www.example.com/products/spring` (em-dash separator, single space either side).
- Given the page is **not** live, the Share Link uses the **Preview URL** instead, and the button's tooltip says **"Page not live — link points to preview"**. The Share Link still works; only the target URL changes.
- Given **both** Live and Preview URL fetches have failed, the Share Link button is disabled with the persistent error state until the `pages.context` page ID changes.
- Keyboard activation follows standard split-button semantics (see §7 FR-006 and §11 UX).

### US-006 — Power user triggers a copy action via keyboard

**As a** keyboard-first marketer,
**I want to** press a single keyboard shortcut to trigger each copy action,
**So that** I don't need to reach for the mouse for a common task.

**Acceptance criteria:**
- Five keyboard shortcuts are bound inside the QuickCopy panel (not global to the Pages editor): one per copy action.
- A **visible legend** at the bottom of the panel displays each shortcut next to its label.
- Activating a shortcut triggers the same behavior (button morph, clipboard write, error state) as a click.
- The chosen shortcut scheme must **not conflict with Sitecore Pages editor's own shortcuts** — conflict verification is a deliverable of the `/architect` stage (see §14 Open Questions).

### US-007 — User toggles light mode

**As a** marketer who prefers light UIs,
**I want to** switch QuickCopy to light mode without leaving the panel,
**So that** the app matches the rest of my workspace.

**Acceptance criteria:**
- A theme switcher control is visible in the panel (location TBD by UI design — header, footer, or corner).
- Toggling between dark and light changes colors, surfaces, and button states immediately, with no panel reload.
- The choice persists across panel reloads via local storage scoped to the app.
- Both themes use Blok tokens throughout — no ad-hoc colors. Light mode is a **first-class** theme, not a "it works but we didn't fuss" afterthought.

## 7. Functional Requirements

- **FR-001 — Panel registration.** The app registers at extension point `xmc:pages:contextpanel` and renders within the Pages editor right sidebar. The panel has no scroll in the default 320px sidebar width with five buttons and the shortcut legend visible.
- **FR-002 — Live context subscription.** The app subscribes to `pages.context` via `client.query('pages.context', { subscribe: true, … })`. On every update, all buttons re-evaluate their enabled/disabled/error state based on the new `pageInfo.id`, `pageInfo.displayName`, and the re-fetched preview/live results for that ID.
- **FR-003 — Preview URL resolution.** When the Preview URL is needed (explicit click, or for Share Link fallback), the app calls `GET /api/v1/pages/{id}/preview-url` via the Marketplace SDK's XMC wrapper. The result is cached per page-item ID for the duration that `pages.context.pageInfo.id` remains that ID.
- **FR-004 — Live URL resolution.** When the Live URL is needed, the app first calls `GET /api/v1/pages/{id}/live` to check publication status. If published, the app composes the Live URL from the Edge host (resolved via `GET /sites/{siteId}/hosts`) plus the `pageInfo.url` slug. If not published, the Live URL button is disabled (tooltip per US-001). The composed Live URL is cached per page-item ID like Preview URL.
- **FR-005 — Share Link formats.** Two formats exist:
  - **Markdown:** `[<pageInfo.displayName>](<resolvedUrl>)` with no trailing whitespace.
  - **Plain text:** `<pageInfo.displayName> — <resolvedUrl>` (em-dash, U+2014, single space either side).
  `<resolvedUrl>` is the Live URL if available, else the Preview URL.
- **FR-006 — Split-button interaction for Share Link.** The Share Link button has two interactive zones: the primary area (default format = Markdown) and a caret area (opens dropdown with Markdown / Plain text). Mouse, touch, and keyboard (Arrow keys, Enter, Escape) must all operate the dropdown per standard split-button semantics. Full keyboard behavior (whether `⌥S` triggers default format vs opens dropdown) is a UX-design deliverable — see §11 and §14.
- **FR-007 — Error state lifecycle.** When an API call or clipboard write fails, the affected button enters a persistent error state (red icon, tooltip with error reason, disabled for clicks). The state clears only when `pages.context.pageInfo.id` changes (user navigates to another page in Pages). Reloading the panel also clears error state. There is no auto-retry, no backoff timer, no manual retry button in v0.1.
- **FR-008 — Success morph.** On successful clipboard write, the clicked button morphs to a "Copied!" state (checkmark icon + "Copied!" label) for exactly 1500ms, then returns to idle. The morph is the sole success feedback — no toast, no sound, no panel-level banner.
- **FR-009 — Clipboard write.** The app uses the clipboard pattern from `products/pageshot/` verbatim. If the pattern doesn't expose a reusable module, factor one out as part of this work so future Marketplace apps inherit it. Clipboard failures trigger the error state per FR-007.
- **FR-010 — Theme switching.** Theme switcher toggles dark/light. State is persisted in iframe-scoped local storage under a key like `quickcopy.theme`. On panel mount, read the key; default to dark if absent. The switcher itself uses a Blok primitive (Toggle or IconButton — final choice in UI design).
- **FR-011 — Keyboard shortcuts.** Five shortcuts, one per button, scoped to the iframe (no global document listeners that would leak into Pages editor). Default scheme: `Alt+L` (Live URL), `Alt+P` (Preview URL), `Alt+I` (Item ID), `Alt+T` (Title), `Alt+S` (Share Link — default format). The scheme is a **strong default**; final binding is confirmed in `/architect` after conflict check with Pages editor hotkeys. If a conflict is found, architect selects a non-conflicting combo and updates this PRD.
- **FR-012 — Shortcut legend.** A compact legend block sits at the bottom of the panel listing all five shortcuts with their labels. Legend uses Blok `Kbd` (or closest equivalent) for visual consistency. The legend is always visible — not hidden behind a help button or collapsed by default.
- **FR-013 — Disabled-state tooltips.** Every disabled button surfaces an actionable tooltip explaining *why* it's disabled and *what the user can do*. Exact copy is defined per US; final wording pass is a UX deliverable.

## 8. Non-Functional Requirements

- **NFR-001 — Performance (cold panel open).** Panel is interactive (five buttons rendered, enabled state decided) within **500ms** of Pages editor opening the sidebar on a warm-cached browser. Cold cache: ≤1000ms acceptable.
- **NFR-002 — Performance (per action).** Click-to-clipboard latency for no-fetch actions (Item ID, Title) ≤50ms. For fetch actions (Preview URL, Live URL), ≤800ms p95 with warm auth, ≤1500ms p95 with cold auth. Share Link depends on underlying URL fetch, same budgets apply.
- **NFR-003 — Accessibility.** All interactive elements are keyboard-operable. ARIA roles and names are set correctly (split-button uses `aria-haspopup`, disabled buttons use `aria-disabled` with tooltip text as `aria-describedby`). Color contrast meets WCAG AA in both dark and light modes. Focus ring is visible.
- **NFR-004 — Browser support.** Chrome, Edge, Firefox, Safari — latest and latest-1. Clipboard API must work on all. The pageshot clipboard pattern is the source of truth; any new browser edge cases discovered during testing feed back into that module.
- **NFR-005 — Resilience.** A single failing API call does not break the panel. Item ID, Title, and Theme Switcher always work even if all HTTP endpoints are down. Buttons that depend on HTTP independently enter error state without affecting the others.
- **NFR-006 — Privacy and data handling.** The app makes no external HTTP calls other than to the Sitecore Marketplace SDK endpoints and Sitecore-hosted APIs resolved through the SDK. No analytics, no third-party scripts, no telemetry in v0.1.
- **NFR-007 — Security.** The app does not persist any content data locally (no page content, no URLs cached to `localStorage`). Theme preference is the only persisted state. No secrets in the client bundle — all auth flows via SDK.
- **NFR-008 — Build and bundle.** Bundle size target: ≤150KB gzipped for the client. Blok is an acceptable dependency (tree-shaken). The app's own code should be ≤30KB gzipped.
- **NFR-009 — Localization readiness.** UI chrome is English in v0.1, but strings are isolated in a single module so a future localization pass is a replacement, not a rewrite.

## 9. Technical Considerations

- **Extension point:** `xmc:pages:contextpanel`. Confirmed per Marketplace SDK catalogue — this is the Pages editor sidebar surface with subscribable `pages.context`.
- **Scaffold pattern:** Follow the **pageshot** scaffold. This is **not** a Content SDK / head-app project. Expected stack: Next.js (for dev tooling) + `@sitecore-marketplace-sdk/client` + Blok — but verify during `/architect` by reading `products/pageshot/` before locking.
- **Data flow:**
  - `pages.context` subscription → single source of truth for `pageInfo.id`, `pageInfo.displayName`, `siteInfo.id`.
  - On `pageInfo.id` change: kick off parallel pre-fetches of `/preview-url` and `/live` so the first click is warm.
  - URL results memoized in a small in-memory map keyed by `pageInfo.id`. Cleared on panel unmount.
- **Clipboard:** Reuse `products/pageshot/` module. If pageshot's clipboard code is inline and not yet a module, factor it out as part of this PRD's implementation. Output location is an architect call (shared package vs duplicated module).
- **Keyboard shortcut conflict check:** Architect must verify `Alt+L/P/I/T/S` do not collide with Sitecore Pages editor hotkeys. If any conflict, architect proposes an alternative (e.g. `Ctrl+Alt+<letter>`), updates this PRD, and documents the change as an ADR.
- **Blok integration:** Components likely needed — `Button`, `IconButton`, `Tooltip`, `Toggle` (theme), `Kbd` (shortcuts), `Menu`/`Dropdown` (split-button). If any primitive is missing, compose from lower-level Blok tokens rather than reaching outside the design system.
- **Live URL composition complexity:** If `/pages/{id}/live` returns only a coarse status flag without the resolved host, the Live URL feature needs three API calls: `/live` (status) → `/sites/{siteId}/hosts` (host) → join with `pageInfo.url` slug. Architect must confirm the call graph and decide caching/pre-fetch policy. This is the single biggest effort risk and should be an ADR candidate.
- **Error propagation:** API failures map to per-button error states, not panel-level errors. Log to `console.error` for developer debugging (removed in prod builds) but surface nothing to the user beyond the button tooltip.

## 10. Data Model Overview

QuickCopy has a minimal in-memory model — no persisted domain data.

```
PageContext (from pages.context, subscribed):
  pageInfo.id: string (GUID)
  pageInfo.displayName: string
  pageInfo.name: string
  pageInfo.url: string (fallback for Live URL slug)
  siteInfo.id: string

PageDerivedState (in-memory, keyed by pageInfo.id):
  previewUrl: string | null | Error
  liveStatus: 'published' | 'unpublished' | null | Error
  liveHost: string | null | Error      (resolved on demand)
  liveUrl: string | null               (composed: liveHost + pageInfo.url slug when status=published)

UIState:
  theme: 'dark' | 'light'              (persisted to localStorage)
  buttonState: map<ButtonId, 'idle' | 'copied' | 'error' | 'disabled'>
```

No database, no external store, no server-side component owned by QuickCopy.

## 11. UX Considerations

### Layout

- **Vertical stack** of five buttons in a single column, ordered by marketer hit frequency: Live URL, Preview URL, Item ID, Page Title, Share Link.
- **Theme switcher** in a consistent location (header or footer corner) — final position is a UI design deliverable.
- **Shortcut legend** at the bottom as a compact row or grid of `label + Kbd` pairs.
- **Panel width:** assume the default Pages editor right-sidebar width (~320–360px). Design must not require horizontal scroll.

### Interaction states per button

| State | Visual | Interactive |
|-------|--------|-------------|
| Idle | Full-color, Blok button primary | Clickable |
| Hover | Blok hover treatment | Clickable |
| Focus | Visible focus ring (WCAG AA contrast) | Clickable |
| Active (pressed) | Blok active | — |
| Copied (success morph) | Checkmark icon + "Copied!" label | Not clickable for 1500ms |
| Disabled | Blok disabled + tooltip on hover | Not clickable |
| Error (persistent) | Red / error accent + ❌ icon + tooltip on hover | Not clickable until context changes |

### Split-button (Share Link) details

- Primary area: default format (Markdown).
- Caret area: opens a dropdown with two menu items (Markdown, Plain text).
- Keyboard on primary: `Enter`/`Space` copies default; keyboard on caret: `Enter` opens menu; `ArrowDown` also opens menu from primary focus.
- See §14 Open Q for shortcut-to-caret vs shortcut-to-default question.

### Theme

- Dark default: Blok dark tokens, neutral greys, accent from Blok primary palette.
- Light: equivalent fidelity. Same component spacing, same icon set, same focus states.
- Switch animation: instant color swap, no fade (keeps the app feeling fast).

### Keyboard legend format

Example rendering:
```
⌨  L  Live URL     P  Preview URL     I  Item ID
   T  Page Title   S  Share Link (Markdown)
```
Exact visual treatment uses Blok `Kbd`. Shown as compact text if Blok doesn't offer that primitive.

## 13. Risks and Mitigations

- **R-001 — Live URL composition is more complex than one API call.**
  *Impact:* pushes effort from XS to S; may require ADR on composition + caching policy.
  *Mitigation:* architect spikes the call graph first. If it exceeds S, consider deferring Live URL to v0.2 and shipping QuickCopy v0.1 with Preview URL + Item ID + Title + Share Link (Preview-only) — stakeholder decision at that point.

- **R-002 — Keyboard shortcuts collide with Pages editor hotkeys.**
  *Impact:* shortcuts are unusable or trigger editor actions unexpectedly.
  *Mitigation:* conflict check is an explicit `/architect` deliverable (see FR-011). If collision, architect changes the scheme and updates this PRD + creates an ADR.

- **R-003 — Clipboard write fails in hardened enterprise browser configurations.**
  *Impact:* silent failure destroys user trust in the whole app.
  *Mitigation:* pageshot's pattern is the proven baseline. The persistent error state (FR-007) converts silent failures into visible failures, which is the correct behavior even if we can't fix the underlying browser block.

- **R-004 — Blok doesn't ship a split-button primitive.**
  *Impact:* build-from-parts cost for Share Link UI.
  *Mitigation:* architect assesses during architecture stage. If missing, compose from Button + Menu + Popover and document as a reusable asset for future Marketplace apps.

- **R-005 — Stale `pageInfo.url` in certain Pages editor states.**
  *Impact:* if used as a fallback for Live URL slug, could leak wrong URLs.
  *Mitigation:* always combine with a fresh `/live` check before exposing. Never expose `pageInfo.url` raw as Live URL.

- **R-006 — Scope creep toward Publish Impact Radar.**
  *Impact:* v0.1 bloats, XS effort balloons, marketer focus dilutes.
  *Mitigation:* hard-coded refusal to add any field beyond the five confirmed buttons. Out of Scope (§5) is treated as authoritative.

- **R-007 — API cache coherency after a publish inside the same session.**
  *Impact:* user publishes a page, then clicks Live URL → still sees disabled because cached `/live` says unpublished.
  *Mitigation:* cache key includes `pageInfo.id` *and* `pageInfo.version`. If Pages editor doesn't update version on publish, architect decides whether to add a manual "refresh" affordance or short-TTL the live-status cache (e.g. 30s).
