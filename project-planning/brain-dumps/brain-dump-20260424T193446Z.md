## 1. Idea Summary

**QuickCopy** is a Sitecore Marketplace **Page Context Panel** app that gives marketers one-click copy buttons for the five pieces of page metadata they paste into Teams/Slack/email/Jira all day long: **Live URL, Preview URL, Item ID, Page Title, and a composite Share Link** (title + URL). No typing, no hunting through the URL bar, no context switch out of the Pages editor.

Lives in `xmc:pages:contextpanel`. Uses the Marketplace SDK `pages.context` query for the bulk of the data, plus two authoritative Agent-API round trips for URL freshness. Built with Blok, dark mode first, keyboard-operable.

## 2. Problem Statement

Marketers spend meaningful time every day sharing page references with teammates — review requests in Slack, bug reports in Jira, release-note entries, status updates. Today this means:

- Hunt for the URL in the browser bar, copy it, often accidentally grabbing the Pages editor chrome URL instead of the actual page URL.
- Open a separate tab to find the Live URL if the page is published.
- Dig into page metadata views to find the Item ID when filing a developer ticket.
- Manually type out "Spring Campaign Landing Page — https://…" combinations for rich-context paste.

Each of these is a tiny annoyance, but together they're hundreds of small context switches per week per marketer. The information is already sitting in `pages.context` and a single Agent-API call. It just needs a one-click surface.

## 3. Target Users

**Primary:** Sitecore marketers and content authors working in the Pages editor. They already have the page open, they already know what they want — they just need the button.

**Secondary:** Marketing ops, project managers, and QA reviewers who work in Pages less often but still need to pass page references to engineering. The composite **Share Link** button is aimed squarely at this group for cross-functional handoffs.

**Not the target:** Developers and Sitecore implementers. They have other tools (Content Editor, XM Cloud deploy pipelines, Experience Edge GraphQL) for the dev/ops metadata (item path, template ID, version numbers). Keeping those out of QuickCopy is a deliberate simplification.

## 4. Proposed Solution

A small Page Context Panel with **five copy buttons**, arranged in a dark-mode-first Blok layout that updates live as the user navigates in Pages. Each button:

- Shows the label and a keyboard shortcut hint (e.g. `Copy Live URL  ⌥L`).
- Morphs on click into a success state ("Copied!" with a checkmark) for ~1.5s, then returns to the default state. **No toast notifications.**
- Is disabled with an explanatory tooltip when the underlying data isn't available (e.g. Live URL button on an unpublished page).

A small keyboard-shortcut **legend** sits at the bottom of the panel so users can discover the shortcuts without opening a separate help view.

A **theme switcher** (dark default, light toggle) sits in the panel header or footer.

## 5. Value Proposition

- **Time saved** — eliminates dozens of daily context switches per marketer.
- **Honesty** — surfaces publication status (Live URL button disabled = not yet live) instead of handing over a broken link.
- **Cross-functional polish** — the Share Link button turns marketer paste into a rich, scannable message (`[Spring Campaign](https://live…)`) that works in Slack, Teams, Notion, and GitHub equally.
- **Dogfood value** — second real-world app (after pageshot) exercising the agentic pipeline on a scope that's truly XS–S. Proves the pipeline handles tiny projects without overhead.

## 6. Core Features

Five buttons, in order of marketer hit frequency:

1. **Copy Live URL** — full absolute URL (including host). Disabled with tooltip "Not published to Edge yet" when `GET /api/v1/pages/{id}/live` reports unpublished.
2. **Copy Preview URL** — full absolute preview URL. Authoritative fetch via Agent API `GET /api/v1/pages/{id}/preview-url` (not `pageInfo.url`, which may be stale or inconsistent).
3. **Copy Item ID** — GUID from `pageInfo.id`. Instant, no round trip.
4. **Copy Page Title** — `pageInfo.displayName`. Instant.
5. **Copy Share Link** — composite of **Page Title + Live URL**, offered in **both formats** (user can pick which — see Decisions):
   - **Markdown:** `[Page Title](https://live-url)` — great for Notion, GitHub, Jira.
   - **Plain text:** `Page Title — https://live-url` — great for email, Slack (auto-linkifies), Teams.
   Falls back to Preview URL if the page isn't live yet (with a tooltip flag so the user knows).

Supporting UI:

- **Theme switcher** — dark default, light toggle, persisted locally (not server-side — marketplace app local storage).
- **Keyboard shortcuts** — one per button (tentative: `⌥L`, `⌥P`, `⌥I`, `⌥T`, `⌥S`). Focus stays in the iframe.
- **Shortcut legend** — compact row or collapsible section showing all shortcuts.

## 7. Decisions

Captured from the rubber-ducky session (user-confirmed, not assumed):

- **Scope is marketer-first.** Dev/ops fields (Item Path, Template ID, Template Name, Version, Site ID) are explicitly **out**.
- **Live URL must be included** and handled honestly — composite endpoint behavior (status check + host lookup) is acceptable added effort.
- **Preview URL** always uses the authoritative Agent API endpoint; does not trust `pageInfo.url`.
- **Multi-site / multi-language ambiguity** is resolved silently by using `pages.context`'s current site + language. No picker UI.
- **Share Link offers BOTH formats** (Markdown + plain text) — mechanism TBD by designer/PM, but both shapes ship in MVP (see Open Questions).
- **Blok is the UI framework.** No ad-hoc Tailwind or custom CSS; use Blok tokens and components.
- **Dark mode is the default.** Light-mode toggle is available but secondary.
- **Keyboard shortcuts** are a first-class feature, with a **legend** visible in the panel (not hidden behind a help button).
- **No toast notifications.** Success feedback is an in-place button state change ("Copied!" morph for ~1.5s).
- **Clipboard pattern is reused from `products/pageshot/`.** Do not reinvent iframe clipboard permission handling.

## 8. Assumptions

Unresolved assumptions (to be validated during PRD / implementation):

- **`pages.context` is always available** inside `xmc:pages:contextpanel` with a valid `pageInfo.id`. True per SDK docs, but edge-case behavior when the Pages editor is loading or the user is viewing a placeholder/layout-only view is not verified.
- **The `/pages/{id}/live` endpoint returns enough signal** to reliably gate the Live URL button. If it only returns a coarse status flag and not the actual live host/slug, implementation will need a second call to `/sites/{siteId}/hosts` and slug-join logic.
- **Blok ships a theme-switcher primitive** or we can compose one from Blok tokens. If not, a lightweight custom switcher is acceptable.
- **Clipboard pattern from pageshot** covers our needs — assumed based on pageshot being a sibling Marketplace Page Context Panel app. Worth a 5-minute read of pageshot's clipboard module before freezing.

## 9. High-Level Technical Considerations

- **Extension point:** `xmc:pages:contextpanel` — confirmed the right surface per SDK catalogue (runs in Pages editor sidebar, has subscribable `pages.context`).
- **Auth model:** The app runs inside the Marketplace iframe. `pages.context` is free; Agent API calls (`/preview-url`, `/live`, `/sites/{siteId}/hosts`) go through the Marketplace SDK `/xmc` wrapper (already auth'd) — **no direct REST calls** unless the SDK has no equivalent.
- **State management:** Minimal. Subscribe to `pages.context` once, re-render buttons on changes, fetch fresh Preview URL + Live status when item ID changes (debounce/cache to avoid hammering the API during rapid Pages navigation).
- **Clipboard:** Reuse pageshot's proven pattern for `navigator.clipboard.writeText` in an iframe. Fallback behavior (execCommand, or user-visible error) inherits from pageshot.
- **Blok design system:** Confirmed as the UI foundation. Components likely needed: `Button`, `IconButton`, `Tooltip`, `Toggle` (for theme), `Kbd` or similar for shortcut hints.
- **Scaffold choice:** **Not Content SDK.** This is a Marketplace App, not a head-app. Follow the pageshot scaffold as reference (likely Next.js + Marketplace SDK client, but verify during `/architect`).
- **Build/deploy:** Marketplace App deployment pattern (SDK-registered iframe). Same as pageshot.

## 10. Risks

- **Live URL composition complexity** — if `/pages/{id}/live` doesn't give us a clean live URL in one response, the feature may require two or three API calls and a slug-join. Could push effort from XS to S. Mitigation: spike this in `/architect`.
- **Clipboard failures in hardened enterprise browser configs** — even with pageshot's pattern, some customer browsers may block clipboard writes entirely. Mitigation: show an explicit error state on the button ("Couldn't copy — check browser permissions") rather than failing silently.
- **Share Link format toggle UX** — offering both Markdown and plain text could bloat the panel or confuse users. Mitigation: keep it as a single compact control (e.g. split-button, or segmented toggle in settings).
- **Keyboard shortcuts conflict with Pages editor** — `⌥L`, `⌥P`, `⌥I`, etc. may collide with Sitecore Pages' own hotkeys. Mitigation: verify during architecture/UI design; fallback to a more unique modifier combo like `Ctrl+Shift+L`.
- **Scope creep toward Publish Impact Radar** — "Is it published?" and "Which hosts?" are already partially needed for Live URL, and could tempt adding more publication info. Mitigation: strict rejection of any field beyond the five confirmed buttons; defer related ideas to backlog #2 (Publish Impact Radar).

## 11. Open Questions

For the PM to resolve during `/create-prd`:

- **Share Link format — how is the user choice exposed?** Segmented toggle in the panel? Setting in the theme switcher menu? Two separate buttons (`Copy Share Link (MD)` / `Copy Share Link (Text)`)? Affects both UX and keyboard-shortcut layout.
- **Default Share Link format** if the user hasn't chosen yet — Markdown (richer) or plain text (more universally compatible)?
- **Keyboard shortcut scheme** — stick with `⌥` + first letter (L/P/I/T/S) or use less-conflict-prone combos? Legend layout at bottom of panel vs. as an overlay?
- **Button state timing** — 1.5s for "Copied!" morph is a guess; should the designer tune this or is it spec-able up front?
- **Light-mode polish** — is light mode genuinely equal to dark mode (full Blok theming), or is it a "it works but we didn't fuss over it" second-class option?
- **Unpublished-page tooltip copy** — exact wording for the disabled Live URL button and the Share Link fallback.
- **Telemetry** — do we want to capture which buttons get used most? (Likely no for MVP; parking this for a later revision.)

## 12. Suggested MVP Scope

**In scope for v0.1 (single short sprint, targeting hours–days of work):**

- The five buttons: Live URL, Preview URL, Item ID, Page Title, Share Link (both formats).
- `xmc:pages:contextpanel` extension point.
- Blok components, dark-mode default, light-mode toggle (theme switcher).
- Keyboard shortcuts with visible legend.
- "Copied!" button state change (no toast).
- Reuse pageshot clipboard pattern.
- Live URL gating via `/pages/{id}/live`.
- Preview URL via authoritative `/preview-url` fetch.

**Deferred (backlog, not v0.1):**

- Dev/ops buttons (Item Path, Template ID, Version).
- Multi-site picker UI (today: use context site silently).
- Language picker UI (today: use context language silently).
- Telemetry on button usage.
- Richer Share Link templating (e.g. custom prefix/suffix).
- "Copy multiple" batch mode.

**Definition of done for MVP:**

- App installs into a tenant and registers as a Page Context Panel.
- All five buttons copy the right data in both a published and unpublished page scenario.
- Dark/light toggle works, preference persists across panel reloads.
- Keyboard shortcuts + legend function; shortcuts don't collide with Pages editor.
- Clipboard write succeeds in the default Sitecore customer browser profile (Chrome/Edge on Windows).
- Button state feedback visible on every successful copy.
